# X402 — Protocol Brief

> **Status:** 🟢 v0 draft — populated from
> github.com/coinbase/x402 specs/x402-specification-v2.md + transports-v2/http.md.
> **Confidence:** high. Spec is published and stable enough to detect; not yet
> widely adopted on e-commerce stores (mostly API monetization use cases).

## What is it?

x402 is an open payment standard built on HTTP's 402 status code. A server gates a resource
behind a 402 response carrying payment requirements; the client signs a payment authorization
(currently stablecoins on EVM/Solana via the "exact" scheme) and retries with a payment
header. Co-developed by Coinbase and the x402 Foundation.

## What does Zephyr check?

A surface-level probe — we do **not** sign payments. We're looking for whether the store has
**any** route advertising x402.

- **Discovery URL:** `https://{origin}/.well-known/x402` (emerging convention; if absent,
  fall back to probing the store's API base if `<link rel="agent-payments" type="x402">`
  is present in the homepage `<head>`).
- **Header probe:** `GET /.well-known/x402` (or the linked URL) — looking for either
  - HTTP 200 with `Content-Type: application/json` containing `accepts: [...]`, OR
  - HTTP 402 with the `PAYMENT-REQUIRED` response header (base64-encoded `PaymentRequired`
    object).

## Pass / fail criteria

| Score | Condition |
|---|---|
| 100 | 402 response with valid `PAYMENT-REQUIRED` header + decoded `x402Version` + `accepts[]` |
| 80  | 200 JSON manifest advertising x402 with `accepts[]` |
| 50  | Endpoint reachable but missing the required headers / fields |
| 0   | Endpoint not found |

## Implementation notes

- The `PAYMENT-REQUIRED` header is base64-encoded JSON. Decode and require
  `x402Version`, `accepts: PaymentRequirements[]`.
- Each `PaymentRequirements` object needs `scheme`, `network`, `amount`, `asset`, `payTo`,
  `maxTimeoutSeconds`.
- Don't follow `PAYMENT-SIGNATURE` flow — this is detect-only.
- Per the v2 spec, expect `PAYMENT-REQUIRED`, `PAYMENT-SIGNATURE`, `PAYMENT-RESPONSE` headers.
